- `_config.yml`：Hexo 主配置。包含站点标题、语言、URL、永久链接、生成器、部署配置、Baidu 主动推送配置等。不要随意公开或改动敏感配置。
- `_config.nexmoe.yml`、`_config.landscape.yml`：旧主题配置或备用配置，目前实际主题是 `archer`。
- `package.json`：项目脚本入口，包含 `build`、`clean`、`deploy`、`server`、`check`、`publish`、`cleanup`。
- `source/`：站点源内容目录，文章、页面、静态资源都在这里。
- `source/_posts/`：博客文章 Markdown 源文件。
- `source/about/index.md`：关于页内容，使用 `layout: about`。

- `source/tags/index.md`：标签页入口，使用 `layout: tags`。

- `source/assets/`：站点图片资源，包括头像、加载图、FPV 软件图标。
- `source/lib/`：站点直接引用的第三方前端库资源。
- `themes/archer/`：当前主题，包含 EJS 模板、CSS、JS、字体、图标和主题配置。
- `themes/archer/_config.yml`：主题配置，控制头像、社交链接、友链、首页标题、副标题、阅读信息、目录、版权、统计、缓存版本等。
- `themes/archer/layout/`：主题页面模板。

- `themes/archer/source/scripts/`：已构建 JS，当前站点直接使用这里的脚本。
- `themes/archer/src/`：主题源 SCSS/JS，如果后续走主题构建流程，应该从这里改再构建。
- `tools/`：项目维护脚本，包含检查、发布、清理。
- `public/`：Hexo 生成产物，被 git 忽略，本地检查和发布时自动生成。

- `node_modules/`：依赖目录，被 git 忽略。
- `自述文件.md`：本文件，项目交接说明，每次修改后必须更新。

## 页面构成框架

### 首页

首页由 `themes/archer/layout/index.ejs` 渲染，主要结构如下：

- 顶部全屏/半屏 intro 区域来自 `themes/archer/layout/_partial/base-background-image.ejs`。
- intro 区显示主标题 `鸽鸽XD的小站` 和副标题 `保持热爱`，配置来自 `themes/archer/_config.yml` 的 `main_title`、`subtitle`。
- intro 背景目前不是图片，而是由 CSS 渐变和浮动标签构成。
- 浮动标签来自站点标签集合，模板会读取 `site.tags`，文章页会优先读取当前文章 tags。
- 正文区域左侧/主区域是文章列表，每篇文章是一个 `article.index-post`。
- 文章摘要由 `index.ejs` 按优先级选择：`excerpt`、`abstract`、截断后的正文。
- 首页右侧/侧栏资料卡来自 `themes/archer/layout/_partial/base-profile.ejs`，显示头像、昵称、签名、社交链接、友链、关于链接。
- 页脚来自主题 footer partial，显示社交链接、HEXO、萌 ICP、访问量统计。

### 文章页

文章页由 `themes/archer/layout/post.ejs` 渲染，主要结构如下：

- 顶部 intro 区显示文章标题、标签、字数、阅读时间、日期、访问量。
- 文章正文输出在 `<article class="article-entry">` 中。
- 版权信息由 `themes/archer/_config.yml` 的 `copyright.enable` 控制。
- 上一篇/下一篇导航由 `post.ejs` 的 `post-paginator` 渲染。
- 评论组件入口存在，但当前主题配置里没有启用具体评论服务。
- 如果文章 front matter 设置 `comments: false`，则不显示评论区域。
- 如果文章 front matter 设置 `mathjax: true`，则文章页会加载 MathJax。

### 关于页

关于页内容在 `source/about/index.md`，front matter 使用：

```yaml
layout: about
```

实际页面模板为 `themes/archer/layout/about.ejs`。当前关于页说明站点诞生时间、GitHub + Hexo 方案、Cloudflare CDN，以及后续内容方向。

### 分类页和标签页

分类页入口是 `source/categories/index.md`，标签页入口是 `source/tags/index.md`。

它们分别使用：

```yaml
layout: categories

```

实际模板位于 `themes/archer/layout/categories.ejs` 和 `themes/archer/layout/tags.ejs`。分类和标签由文章 front matter 自动生成。

### 404 页

404 入口是 `source/404.md`，使用 `layout: 404`。实际模板是 `themes/archer/layout/404.ejs`。

## 文章格式约定

文章放在 `source/_posts/`，使用 Markdown。典型 front matter：

```yaml
---
title: 文章标题
tags:
  - 标签1
  - 标签2
categories:
  - 分类名
abbrlink: stable-url
date: 2026-06-20 14:22:37
ai: true
---
```

正文风格：

- 常见结构是一级标题、介绍段、`---` 分隔线、多个二级标题。
- 教程类文章尽量按小白流程写，说明“输入什么、看到什么、下一步做什么”。
- 文章里可以使用代码块、列表、HTML 嵌入。
- `abbrlink` 决定最终 URL，例如 `abbrlink: test-page` 会生成 `/posts/test-page.html`。
- 如果用户提供文章内容，AI 只修改或发布，不要自动添加 AI 标志。

## AI 标志约定

首页右侧小红色 `AI` 标志是用户约定的作者标识。

- 如果文章主要由 AI 从零撰写，在文章 front matter 中加入：

```yaml
ai: true
```

- `themes/archer/layout/index.ejs` 会判断 `_post.ai === true`，然后渲染：

```html
<span class="ai-post-badge" title="AI assisted">AI</span>
```

- `themes/archer/source/css/style.css` 中 `.ai-post-badge` 控制这个标志的样式。
- 如果文章是用户自己写的，AI 只是修改、排版、修错或发布，不要添加 `ai: true`。
- 如果一篇文章是混合情况，按主要作者判断；不确定就先问用户。

## 发布和检查流程

优先使用项目自带脚本：

```bash
tools/check.sh
tools/publish.sh "提交信息"
tools/cleanup.sh
```

`tools/check.sh` 做的事：

- 执行 `git diff --check`，检查空白和格式问题。
- 检查是否误提交 Vim swap 文件。
- 执行 `hexo clean` 和 `hexo generate`。
- 检查 `public/index.html`、`public/sitemap.xml`、`public/baidusitemap.xml`、`public/content.json` 是否存在。
- 检查 `themes/archer/_config.yml` 的 `source_version` 是否出现在生成后的首页 CSS 引用里。

`tools/publish.sh "提交信息"` 做的事：

- 先运行 `tools/check.sh`。
- 如果有源码改动，会 `git add -A` 并提交。
- 推送当前源码分支到远端，通常是 `source`。
- 执行 `hexo deploy`，把 `public/` 生成内容推到 `master`。
- 脚本退出时会自动调用 `tools/cleanup.sh`。

`tools/cleanup.sh` 做的事：

- 停止 4000 端口上的本地 Hexo 预览。
- 停止当前工作区相关 Hexo 命令。
- 检查 Hexo、Node、Git 相关残留进程。

如果系统没有全局 Node/npm，可以使用 Codex 自带 Node 路径：

```bash
PATH="/Users/gegexd/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/bin:$PATH" tools/check.sh
```

## 发布注意事项

- Baidu 主动推送可能返回 `401 site error`，这不代表 GitHub Pages 部署失败。
- 发布后线上页面可能短暂 404，通常等几十秒再请求即可。
- 发布完成后用 `curl -I https://gegexd.com/posts/xxx.html` 验证线上状态。
- 如果开过本地预览，发布后必须停止预览进程。
- 只改根目录文档、未影响站点生成内容时，可以只提交并推送 `source`，不一定要部署 `master`。
